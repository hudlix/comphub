Stay compliant and in control — without replacing your stack.

CompHub is an open-source control plane for engineering teams. It unifies vulnerability SLAs, delivery cadence, service health, observability, and reporting into one launcher of independently enableable workspaces — connected to your tools through pluggable adapters.

The problem

Compliance and control are scattered across ten tabs.

SLAs live in the scanner. Delivery lives in the PM tool. Health lives in dashboards. Context lives in someone’s head. CompHub sits on top and reads from all of them — so the state of your apps is one launcher away, not a manual roll-up every Friday.

Workspaces

Six workspaces. Enable only what you need.

Each one is an isolated module, switchable per deployment and per user. The launcher and settings are the shared shell.

Delivery

Epic → Story → Task from your PM provider, with a lifecycle stage derived from a configurable rule — never stored.

Vulnerabilities

Scanner findings on a remediation SLA clock. Scan-and-remediate proposes a PR for review — it never mutates a repo on its own.

Observability

Reporting

How it works

From zero credentials to a live control plane.

CompHub boots on mock adapters, so you can run the whole thing before connecting anything real.

Run in mock mode

Clone, install, and start. Every workspace runs on demo data — no secrets required.

Connect adapters

Swap in a PM, scanner, or SCM provider through config. Vendor-neutral by design.

Enable workspaces

Turn on the workspaces your team needs, per deployment and per user.

Standardize repos

One PR installs the .comphub/ structure. Resume any session with /comphub.

Continuity

Work that survives the person who did it.

  • Git is the system of record. Session memory is committed in your repos under .comphub/, reviewed in the same PR as the work.
  • CompHub reads and launches it. The app standardizes the structure and resumes sessions; it doesn’t write your logs.
  • A fresh agent can pick up cold. Hand an incident or a half-built feature to anyone — or any model — with full context.
$ /comphub log "debug payment retry timeout"
✓ wrote .comphub/sessions/2026-06-25-payment-retry.md
# committed alongside the fix in PR #142

$ /comphub resume
↻ loading last session · checkout-service
✓ context restored — 3 decisions, 2 open threads
  ↳ retry backoff still unverified under load

Architecture

Built to be extended, not forked.

Pluggable adapter ports

PmProvider, ScannerProvider, ScmProvider — concrete vendors are just example implementations.

One UI app, one API app

A modular monorepo. Each workspace is cleanly isolated, so any one can be promoted to its own deployable later.

Human-in-the-loop

Mutations default to a dry-run and a PR. Nothing changes your repos without a review.

Policy as configuration

SLA windows, the lifecycle stage rule, enablement — all editable per deployment, not hardcoded.

Git is the system of record

Continuity memory lives in your repos, not a CompHub database you can’t leave.

Open-core ready

The enablement model is a clean seam. Enterprise concerns stay separable without bending the open core.

FAQ

The usual questions.

Is CompHub free?

Yes. The core is open source under Apache-2.0 and you self-host it. A paid enterprise tier (SSO, fine-grained RBAC, multi-tenant, audit, advanced reporting) is planned later as open-core — the core stays open.

Do I have to use Jira or GitHub?

No. Integrations are pluggable adapter ports, and a mock adapter ships on by default. Jira, GitHub, GitLab, and scanners are example adapters; you can add your own.

Where does my data live?

In your deployment. Continuity session memory lives in your own git repos under .comphub/ — there’s no CompHub database you can’t walk away from.

Does it replace my existing tools?

No — that’s the point. CompHub reads from the tools you already use and gives you one place to see and act on compliance and delivery state.

What’s the status?

Early. The product site and specs are public now; the application scaffold is the next milestone. Follow along or contribute on GitHub.
